## Day1 — 智能体基础

`📅 2026年7月13日`

### 1. 什么是智能体

> 能够通过传感器感知其所处环境，并自主地通过执行器采取行动以达成特定目标的实体。

### 2. 智能体分类

| 类型 | 特点 |
|------|------|
| 反应式智能体 | 直接对环境刺激做出响应，无内部模型 |
| 规划式智能体 | 具备内部世界模型，能进行目标规划 |
| 混合式智能体 | 结合反应式与规划式的特点 |

### 3. Workflow vs Agent

> **Workflow** 是让 AI 按部就班地执行指令；**Agent** 则是赋予 AI 自由度去自主达成目标。

### 4. 智能体的运行机制

智能体的行动会引起环境 (Environment) **状态变化 (State Change)**，环境随即产生一个新的 **观察 (Observation)** 作为反馈。新的观察又会在下一轮循环中被感知系统捕获，形成一个持续的 **感知 → 思考 → 行动 → 观察** 闭环。智能体正是通过不断重复这一循环，逐步推进任务，从初始状态向目标状态演进。

```
┌─────────────────────────────────────────────────┐
│                                                 │
│   Agent ──► Planning ──► Thought ──► Action     │
│     ▲                                │          │
│     │                                ▼          │
│  Observation ◄── State Change ◄── Environment   │
│                                                 │
└─────────────────────────────────────────────────┘
           图1.5 智能体与环境交互的基本循环
```

### 5. 智能体的核心构成

- **Perception（感知）** — 从环境中获取信息
- **Thought（思考）** — 智能体内部决策的"快照"。以自然语言形式阐述如何分析当前情境、回顾上一步观察结果、进行自我反思与问题分解，并规划出下一步的具体行动
- **Action（行动）** — 基于思考后对环境施加的具体操作，通常以函数调用的形式表示

```python
# 示例
Thought: 用户想知道北京的天气。我需要调用天气查询工具。
Action:  get_weather("北京")
```

### 6. 开发中用到的工具

| 工具 | 说明 |
|------|------|
| **wttr.in** | 免费的天气查询服务，以 JSON 格式返回指定城市的天气数据 |
| **tavily-python** | 强大的 AI 搜索 API 客户端，用于获取实时的网络搜索结果。[官网](https://www.tavily.com/) |

## Day2 — 大模型基础

`📅 2026年7月14日`

### 从 N-gram 到 Transformer：语言模型的演进之路

| # | 模型/技术 | 核心思想 | 解决的问题 |
|---|----------|---------|-----------|
| 1 | **N-gram（马尔可夫假设）** | 一个词的出现概率只与前面有限的 N 个词有关 | 用有限上下文窗口近似语言概率 |
| 2 | **前馈神经网络语言模型** | 构建语义空间（词向量），学习从上下文到下一个词的预测 | 引入连续向量表示，用余弦相似度衡量语义 |
| 3 | **循环神经网络 (RNN)** | 每步读取当前输入 + 上一刻隐藏状态，生成新隐藏状态 | 为网络增加"短期记忆"能力 |
| 4 | **LSTM** | 引入**细胞状态 (Cell State)** + **门控机制**（遗忘门、输入门、输出门） | 解决 RNN 的梯度消失/爆炸问题 |
| 5 | **Transformer** | 使用**注意力 (Attention)** 机制捕捉序列内依赖关系 | 实现真正意义上的并行计算 |
| 6 | **前馈神经网络** | 从注意力聚合后的信息中提取更高阶特征 | 增强特征表达能力 |
| 7 | **残差连接 (Add)** | 将子模块的输入**直接加到输出**上 | 解决深度网络的梯度消失问题 |
| 8 | **层归一化 (Norm)** | 对单个样本的所有特征归一化（均值 0，方差 1） | 解决内部协变量偏移 (Internal Covariate Shift) |
| 9 | **分词 (Tokenization)** | 将文本序列转换为数字序列 | 让模型能处理自然语言文本 |

### 10. Hugging Face Transformers

> 一个强大的开源库，提供标准化接口来加载和使用数以万计的预训练模型。

### 11. 代表性开源模型

- **Meta Llama** — Meta 出品，社区生态最丰富
- **Mistral AI** — 法国团队，小模型性能出色
- **Qwen（通义千问）** — 阿里出品，中文能力领先

### 12. 模型幻觉 (Hallucination)

> 大语言模型生成的内容与客观事实、用户输入或上下文信息相矛盾，或者生成了不存在的事实、实体或事件。

**幻觉的本质：** 模型在生成过程中，过度自信地合成信息，而非准确地检索或推理。

**解决方向：** RAG（检索增强生成）、多步推理与验证、外部工具调用。

## Day3 — ReAct 范式

`📅 2026年7月15日`

### 1. 什么是 ReAct

> **ReAct (Reasoning and Acting)** — 一种将”思考”和”行动”紧密结合的范式，让智能体**边想边做，动态调整**。

核心思想：思考与行动相辅相成。思考指导行动方向，行动的结果反过来修正思考。智能体不断重复 **Thought → Action → Observation** 的循环，将新的观察结果追加到历史记录中，形成不断增长的上下文，直到在 Thought 中认为找到了最终答案。

### 2. ReAct 的协同效应

| 维度 | 作用 |
|------|------|
| **推理 (Reasoning)** | 使行动更具目的性，避免盲目调用工具 |
| **行动 (Acting)** | 为推理提供事实依据，减少幻觉 |

### 3. 执行流程

```
用户提问
  │
  ▼
Thought: 分析问题，决定下一步行动
  │
  ▼
Action: 调用工具并获取结果
  │
  ▼
Observation: 工具返回真实数据
  │
  ▼
Thought: 根据结果继续推理...
  │
  ├─ 信息不足 ──► 回到 Action（循环）
  │
  └─ 信息充分 ──► Final Answer
```

### 4. 实践要点

本课程通过**手动搭建 ReAct Agent** 体验了完整流程，同时对 `LLMClient` 和 `search()` 工具进行了封装：

- 封装 LLMClient — 统一接口层，方便切换模型
- 封装 search() — 赋予 Agent 获取实时信息的能力

## Day4 — 高级智能体范式

`📅 2026年7月16日`

### 1. Plan-and-Solve（先规划，后执行）

> 如果说 **ReAct** 像一位经验丰富的侦探，根据现场蛛丝马迹一步步推理、随时调整方向；那么 **Plan-and-Solve** 更像一位建筑师，先绘制完整蓝图，再严格按图施工。

#### 工作原理

| 阶段 | 任务 | 说明 |
|------|------|------|
| **规划阶段 (Planning)** | 分解问题，制定行动计划 | 接收完整问题，将问题拆解为分步骤计划（一次 LLM 调用） |
| **执行阶段 (Solving)** | 逐步执行计划 | 严格按计划逐一执行，每步可以是独立 LLM 调用或对上一步结果的加工 |

#### 状态管理

规划器生成行动蓝图后，执行器 (Executor) 负责逐一完成任务。执行器不仅调用 LLM 解决每个子问题，还承担**状态管理**的职责——记录每一步执行结果，并将其作为上下文提供给后续步骤。

执行器的提示词需包含以下关键信息：

| 信息 | 作用 |
|------|------|
| 原始问题 | 确保模型始终了解最终目标 |
| 完整计划 | 让模型了解当前步骤在整个任务中的位置 |
| 历史步骤与结果 | 提供已完成的工作，作为当前步骤的直接输入 |
| 当前步骤 | 明确指示模型现在需要解决的具体任务 |

---

### 2. Reflection（反思式智能体）

> 核心思想：为智能体引入一种**事后的自我校正循环**，使其能够像人类一样审视自己的工作、发现不足、并迭代优化。相当于给智能体加了一个”内部评审员”。

#### 执行逻辑

```
执行初稿 ──► 自我反思 ──► 发现不足 ──► 优化迭代 ──► 再次反思 ──► ... ──► 满意输出
```

#### 优缺点

| | 说明 |
|---|------|
| **优点** | 解决方案质量高、可靠性强、适合复杂任务 |
| **缺点** | 模型调用成本增加、任务延迟高、提示词工程复杂 |

---

### 3. 三种范式如何选择

| 范式 | 适用场景 | 类比 |
|------|---------|------|
| **ReAct** | 任务充满不确定性，需要与外部 API 或网页频繁交互 | 侦探 — 边走边查 |
| **Plan-and-Solve** | 逻辑路径清晰，侧重内部推理和步骤分解 | 建筑师 — 先画蓝图 |
| **Reflection** | 对最终结果的质量和可靠性有极致要求 | 作家 — 反复打磨 |

> **组合使用：** 实际项目中三种范式往往组合使用。例如用 Plan-and-Solve 做宏观规划，ReAct 做每一步的具体执行，Reflection 在最终输出前做质量审查。

## Day5、6— 框架开发实践
`📅 2026年7月17日`
### 1. 框架优势

- 提升代码复用和开发效率
- 实现核心组件的解耦与可扩展性
- 标准化复杂的状态管理
- 简化可观测性与调试过程

### 2.主流框架

1. AutoGen:  以对话驱动协作，将复杂的任务解决流程，映射为不同角色的智能体之间的一系列自动化对话。定义智能体的核心在于编写高质量的系统消息 (System Message)。系统消息就像是给智能体设定的“行为准则”和“专业知识库”，它精确地规定了智能体的角色、职责、工作流程，甚至是与其他智能体交互的方式。一个精心设计的系统消息是确保多智能体系统能够高效、准确协作的关键。白话：“对话工作室”。

2. AgentScope: **工程化优先的多智能体平台**，**采用消息驱动的架构设计**，内置了分布式部署、容错恢复、可观测性、消息解耦、异步并发等企业级特性。白话：“智能体操作系统（分布式系统 ）”，为开发者提供了从开发、测试到部署的全生命周期支持。

  - 内置了一个**消息中心 (MsgHub)**，它是整个消息驱动架构的中枢。MsgHub 不仅负责消息的路由和分发，还集成了持久化和分布式通信等高级功能

3. CAMEL: 目标是探索如何在最少的人类干预下，让两个智能体通过“角色扮演”自主协作解决复杂任务。
   
   - 核心是"角色扮演 + 引导性提示"——一段精心设计的初始指令，让两个智能体自己跑完整个协作流程，不需要复杂的流程控制。
   - 一个扮演 **“AI 用户” (AI User)**，负责提出需求、下达指令和构思任务步骤；另一个则演 **“AI 助理” (AI Assistant)**，负责根据指令执行具体操作和提供解决方案。
   - 轻架构，重提示。通过精心设计的初始提示就能实现高质量的智能体协作。这种自然涌现的协作行为，往往比硬编码的工作流更加灵活和高效。
   - 局限性：对提示词工作高度依赖、协作规模限制
   - CAMEL 代表了一种独特而优雅的多智能体协作范式。它通过"以人为本"的角色扮演设计，将复杂的系统工程问题转化为直观的人际协作模式。

4. LangGraph: 代表了智能体框架设计的一个全新方向。与前面介绍的基于“对话”的框架（如 AutoGen 和 CAMEL）不同，LangGraph 将智能体的执行流程建模为一种**状态机（State Machine）**，并将其表示为**有向图（Directed Graph）**。

   - 在这种范式中，图的**节点（Nodes）** 代表一个具体的计算步骤（如调用 LLM、执行工具），而**边（Edges）** 则定义了从一个节点到另一个节点的跳转逻辑。这种设计的革命性之处在于它天然支持循环，使得构建能够进行迭代、反思和自我修正的复杂智能体工作流变得前所未有的直观和简单。
   - 组建工作流模式，工作是节点，流是边，自定义连接节点和边。把智能体的执行流程画成一张有向图。
   - 实例：问答助手：理解->搜索->回答。
   - 优势：流程可控、高度模块化（节点）。
   - 缺点：需要开发更多代码、调试困难。

### 3.总结

| AutoGen| 将复杂的协作抽象为一场由多角色参与的、可自动进行的“群聊”，其核心在于“以对话驱动协作”。|
|-|-|
| AgentScope | 着眼于工业级应用的健壮性与可扩展性，为构建高并发、分布式的多智能体系统提供了坚实的工程基础。|
| CAMEL | 以其轻量级的“角色扮演”和“引导性提示”范式，展示了如何用最少的代码激发两个专家智能体之间深度、自主的协作。|
| LangGraph  | 回归到更底层的“状态机”模型，通过显式的图结构赋予开发者对工作流的精确控制，尤其是其循环能力，为构建可反思、可修正的智能体铺平了道路。|

**总而言之：智能体并非只有一种构建方式，需要根据不同项目灵活使用不同的框架**


## Day7— 构建你的agent框架

`📅 2026年7月20日

### 1. 框架架构设计

>原则：原则：分层解耦（核心层、智能体层、工具层各司其职）、职责单一（每个模块只做一件事）、接口统一（所有组件通过标准接口交互）。项目结构分为 core（消息、配置、LLM接口）、agents（各种范式实现）、tools（工具注册与执行）三大模块。

### 2. 多模型支持

> LLM 层通过继承实现多供应商适配。基类 HelloAgentsLLM 封装了 OpenAI 兼容接口，子类只需覆盖模型名和 base\_url 即可接入新供应商（如 DeepSeek、ModelScope 等）。

> 本地模型方面，框架支持 VLLM 和 Ollama 两种部署方案。VLLM 适合高吞吐生产场景，Ollama 则以便捷的本地体验见长，二者均暴露 OpenAI 兼容端点，框架无需额外适配即可调用。

### 3. 集成四种范式

1. SimpleAgent 是最基础的对话模式，直接将用户输入交给 LLM 并返回结果，无工具调用、无多步推理，适合简单问答场景。

2. ReActAgent 实现"推理-行动"循环：每步先 Thought（分析问题），再 Action（调用工具或给出最终答案），工具返回 Observation 后进入下一轮，直到得出 Finish 结论或达到最大步数。

3. ReflectionAgent 引入自我反思机制：先执行得到初始输出，再让 LLM 以"评审者"视角审视输出质量并提出改进建议，然后根据建议迭代优化，循环若干轮后输出最终版本。

4. PlanAndSolveAgent 采用"先规划后执行"策略：第一步让 LLM 将复杂任务分解为子步骤列表，然后逐步执行每个子任务并收集结果，最终汇总为完整答案。适合多步骤、有依赖关系的复杂问题。

### 4. 工具模块

> 工具层由 Tool 基类和 ToolRegistry 注册表组成。Tool 基类规定每个工具必须有 name、description 和 execute 方法；ToolRegistry 负责注册、检索和统一调用，并向 LLM 生成工具描述文本供其选择。

> 高级搜索工具 MyAdvancedSearchTool 以类的方式构建，内部整合 Tavily 和 SerpAPI 两个搜索源，自动检测可用 API 密钥，将多源结果合并格式化后返回。类方式适合需要维护状态（客户端实例、配置）的复杂工具。

> 工具链（ToolChain）支持多工具顺序编排：定义步骤序列，每步的输出可作为下步的输入变量，由 ToolChainManager 统一管理和执行，实现类似流水线的组合调用。

> 异步执行器（AsyncToolExecutor）基于线程池和 asyncio，支持并行调用多个工具，适合多工具独立任务场景，显著缩短总耗时。
